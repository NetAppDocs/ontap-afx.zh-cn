= AFX 存储系统 SVM 的附加管理
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
除了典型的 AFX SVM 管理之外，您可能还需要根据您的环境执行其他任务。大多数附加任务都可以使用系统管理器执行，但在某些情况下您可能需要使用 CLI。


NOTE: 所描述的ONTAP功能和管理对于运行 Unified ONTAP 的AFX 存储系统和AFF或FAS系统来说是通用的。其中包含指向相关 Unified ONTAP文档的链接（视情况而定）。



[[storage-management-and-performance]]
== 存储管理和性能

您可以在 AFX 部署中配置和使用多种可选的存储管理和性能功能。



[[ontap-nas-storage-management]]
=== ONTAP NAS 存储管理

网络连接存储 (NAS) 提供专用文件存储，可由网络中的多个客户端访问。ONTAP 支持多种 NAS 协议。详细了解 https://docs.netapp.com/us-en/ontap/nas-management/index.html["NAS存储管理"^]。



[[ontap-flexcache-volumes]]
=== ONTAP FlexCache卷

FlexCache 是 ONTAP 远程缓存功能。它使数据更接近客户端，从而提高访问性能并降低成本。创建一个 FlexCache 卷，最初只从原始文件系统中复制元数据，简化了文件分发并减少了 WAN 流量。详细了解 https://docs.netapp.com/us-en/ontap/flexcache/index.html["ONTAP FlexCache卷"^]。



[[ontap-flexgroup-volumes]]
=== ONTAP FlexGroup卷

FlexGroup 卷由自动透明地共享流量的多个成员卷组成。FlexGroup 卷具有多种优点，包括高性能和简化管理。详细了解 https://docs.netapp.com/us-en/ontap/flexgroup/creation-workflow-task.html["FlexGroup卷设置"^]。



[[cross-file-sequential-read]]
=== 跨文件顺序读取

交叉文件顺序读取（CFSR）是从 ONTAP 9.19.1 开始的 AFX 系统上可用的一项功能，可提高视频编辑和相关应用程序的性能。对于将视频帧存储为单个顺序文件（通常每个文件 50 至 250 MB）的工作流，CFSR 基于顺序文件名模式智能地预测并预取下一个文件。可以预取多达 10 个文件或 500 MB 的数据，大大减少了读取延迟，并确保以每秒 30 帧的速度平滑播放 4K/8K 原始视频。

默认情况下，CFSR 在运行 ONTAP 9.19.1 或更高版本的 AFX 系统上处于启用状态，但您需要在单个卷上显式启用它。存储管理员可以使用 ONTAP CLI 或等效的 REST API 调用来配置该功能。使用 CLI，您可以启用 CFSR 作为创建新卷的一部分。您还可以使用以下命令更新现有卷：

`volume modify -volume <volume_name> -aggressive-readahead-mode [cross_file_sequential_read|none]`

.示例
[source, bash]
----
volume modify -volume myvolume -aggressive-readahead-mode cross_file_sequential_read
volume show -volume myvolume -fields aggressive-readahead-mode
----
CFSR 功能与 FlexVol 和 FlexGroup 卷兼容，并且与文件扩展名无关，支持十进制和十六进制文件名编号方案。对于 NFSv4.x 协议，CFSR 与多协议 I/O (MPIO) 读取功能集成，通过直接从内存提供后续读取来进一步提高性能。CFSR 仅在 AFX 系统上提供，对支持的卷有限制。以下卷类型不支持此功能：

* 启用了 FabricPool 的卷
* FlexCache 端点卷
* SnapMirror 目标卷


任何其他 ONTAP 平台（FAS、AFF 或 ASA）或云部署（如 Cloud Volumes ONTAP）均不支持 CFSR。它可以使用启用或禁用 GDD 的 FlexGroup 卷进行操作。但是，由于视频帧文件通常比 GDD 的最小段大小小得多，因此 GDD 无法有效地跨 FlexGroup 组分对这些文件进行条带化。这意味着 GDD 对 CFSR 的文件预取行为的直接影响最小。



[[data-protection]]
== 数据保护

您可以在 AFX 部署中配置和使用多种可选的数据保护功能。



[[consistency-groups]]
=== 一致性组

一致性组是作为一个单位管理的卷的集合。详细了解 https://docs.netapp.com/us-en/ontap/consistency-groups/index.html["ONTAP 一致性组"^]。



[[snaplock]]
=== SnapLock

您可以通过在卷级别将文件转换为一次写入多次读取 (WORM) 状态来保护文件。SnapLock 支持两种模式。合规模式可确保在其保留期到期之前无法删除文件，这解决了政府或行业的特定要求。企业模式允许特权用户在保留期到期之前删除文件。详细了解 https://docs.netapp.com/us-en/ontap/snaplock/index.html["ONTAP SnapLock"^]。



[[storage-efficiency]]
== 存储效率

ONTAP 提供多种存储效率功能，可帮助您优化存储使用并降低成本。



[[global-deduplication]]
=== 全局重复数据删除

https://docs.netapp.com/us-en/ontap/concepts/deduplication-concept.html["重复数据删除"^] 是一种通过识别和删除重复数据块来降低物理存储要求的  https://docs.netapp.com/us-en/ontap/volumes/deduplication-data-compression-efficiency-concept.html["ONTAP 存储效率"^] 功能。传统的重复数据删除在卷或聚合级别上运行。这种现有功能为所有 ONTAP 平台的客户提供了显著的存储效率。

从 ONTAP 9.19.1 开始，AFX 通过将重复数据删除扩展到整个存储可用区 (SAZ)，引入了新的操作范围。全局重复数据删除以卷级重复数据删除为基础，自动消除整个 SAZ 中各卷的重复数据块。全局重复数据删除会话会自动触发，从而确保无需手动干预即可实现最佳节省。

默认情况下，全局重复数据删除在 AFX A1K 系统上启用，并在后台透明运行，对性能的影响最小。无需功能许可，该功能与现有的存储效率技术（包括压缩和压实）无缝集成。通过全局重复数据删除实现的节省通过存储效率指标进行报告。此功能对于跨多个卷频繁共享或复制数据的应用程序工作负载尤其有益。



[[security]]
== 安全性

您可以在 AFX 部署中配置和使用多种可选的安全功能。



[[fpolicy]]
=== FPolicy

FPolicy 是一个文件访问通知框架，用于监视和管理存储虚拟机 (SVM) 上的文件访问事件。您可以使用 FPolicy 创建策略来定义要监视的文件操作，并根据您定义的条件选择性地阻止这些操作。FPolicy 通常用于安全审计、合规性和数据治理。了解更多 https://docs.netapp.com/us-en/ontap/nas-audit/two-parts-fpolicy-solution-concept.html["ONTAP FPolicy 解决方案"^]。
